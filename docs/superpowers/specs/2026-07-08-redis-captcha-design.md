# 登录与注册验证码 Redis 化设计

日期：2026-07-08

## 目标

将登录、注册验证码从 HttpSession 迁移到 Redis，提供明确的两分钟有效期、前端倒计时、自动刷新和后端防刷能力，同时保持现有 Thymeleaf 登录/注册流程。

## 当前状态

- `/captcha` 每次请求都会生成验证码并写入 HttpSession。
- 验证码没有独立 TTL，实际生命周期跟随 Session。
- 登录和注册成功验证后才删除验证码。
- 验证码图片可以无限点击刷新。
- 项目已有 Redis、`StringRedisTemplate` 和基于 Redis 的 `RateLimitInterceptor`。
- 现有限流 key 按 IP 或用户生成，但未包含接口维度，可能导致不同接口共享计数。

## 选定方案

采用“Session ID + Redis”方案。Session ID 只作为匿名验证码挑战的关联标识，验证码答案、失败次数、刷新冷却和频率计数均存入 Redis。

### Redis Key

- `captcha:code:{sessionId}`：验证码答案，TTL 120 秒。
- `captcha:fail:{sessionId}`：当前验证码错误次数，TTL 与验证码同步，最多 5 次。
- `captcha:cooldown:{sessionId}`：会话刷新冷却，TTL 5 秒。
- `captcha:rate:{endpoint}:{clientIp}`：IP 获取频率，60 秒最多 10 次。

验证码仅存字符串，不进入 Spring Cache 的通用 JSON 缓存区域。

## 后端流程

### 获取验证码

1. `/captcha` 只接受 GET，不接收或传输密码。
2. 根据可信代理头和远端地址确定客户端 IP。
3. 检查会话刷新冷却；不足 5 秒返回 HTTP 429，并返回剩余等待秒数。
4. 检查 `captcha` 接口的 IP 频率；60 秒超过 10 次返回 HTTP 429。
5. 生成新验证码，覆盖 `captcha:code:{sessionId}`，TTL 120 秒。
6. 删除旧失败计数，创建 5 秒冷却 key。
7. 返回 PNG，并通过响应头提供有效期 120 秒，禁止浏览器缓存。

### 校验验证码

1. 输入为空、Redis key 不存在或已过期时校验失败，并返回明确的“验证码已过期，请刷新”。
2. 输入正确时删除验证码和失败次数，确保验证码只能成功使用一次。
3. 输入错误时原子增加失败次数。
4. 连续错误达到 5 次时删除验证码，要求重新获取。
5. 登录和注册共用验证码服务，不再直接访问 HttpSession 属性。
6. Redis 临时异常时降级到短生命周期 Session 验证码；防刷计数不可用时记录告警，但不让登录系统完全不可用。

## 限流修正

现有 `RateLimitInterceptor` 的 key 改为包含接口标识，例如：

`rate_limit:{HTTP_METHOD}:{URI}:{keyType}:{IP或userId}`

这样登录、注册、验证码及聊天接口不会因为使用同一 IP 或用户而共享限流计数。

验证码获取可使用专用服务返回图片与限流结果，避免通用拦截器难以向图片请求返回精确冷却时间的问题。

## 前端交互

登录页和注册页采用相同规则：

1. 页面初始不请求验证码，显示“请先输入账号和密码”。
2. 账号和密码均非空后，自动加载第一张验证码。
3. 验证码加载成功后显示 `有效期 02:00`，每秒递减。
4. 倒计时归零时：
   - 账号和密码仍非空：自动请求新验证码并从 `02:00` 重新计时；
   - 任一字段为空：停止计时与刷新，恢复初始禁用状态。
5. 用户手动点击验证码时执行同样的字段检查。
6. 5 秒冷却期内再次点击时不替换当前图片，显示“请在 N 秒后刷新”。
7. 收到 429 时显示后端返回的频率提示，不启动错误的倒计时。
8. 自动刷新或手动刷新成功后清空验证码输入框，避免提交旧验证码。
9. 页面隐藏或离开时清理计时器，回到页面后根据截止时间校正剩余秒数，避免后台标签页计时漂移。

前端仅用账号和密码是否为空控制交互；密码不会随验证码图片请求发送。后端安全性始终由 Redis 冷却与 IP 限流保证，不能依赖前端禁用。

## 错误处理

- 未填写账号或密码：前端不发送请求并显示提示。
- 验证码过期：登录/注册页显示“验证码已过期，请刷新”，随后获取新验证码。
- 验证码错误：显示现有错误提示；达到 5 次时提示验证码已失效并刷新。
- 获取过频：返回 429 和可展示的信息；原验证码在自身 TTL 内仍可使用。
- Redis 异常：记录服务端告警，启用 Session 降级路径。

## 测试范围

### 后端

- 验证码写入 Redis 且 TTL 接近 120 秒。
- 新验证码覆盖旧验证码。
- 正确验证码只能成功一次。
- 错误 5 次后验证码失效。
- 5 秒内刷新返回 429。
- 同一 IP 60 秒超过 10 次返回 429。
- 不同接口的限流 key 相互独立。
- Redis 不可用时 Session 降级路径可工作。

### 前端

- 账号或密码为空时不请求验证码。
- 两个字段填写后只加载一次初始验证码。
- 倒计时从 02:00 正确递减。
- 到期后自动刷新并重置倒计时。
- 刷新后验证码输入框被清空。
- 字段清空后停止自动刷新。
- 429 提示和冷却剩余时间正确显示。

### 部署验证

- Maven 测试和打包通过。
- Redis 中验证码 key 的 TTL 正确。
- 登录页、注册页验证码获取和刷新正常。
- systemd 重启后服务正常启动。
- Nginx 域名、HTTP、HTTPS 和后台页面返回正常状态。

## 不在本次范围

- 短信或邮件验证码。
- 无 Session 的独立 captcha token 协议。
- 修改验证码图片风格或字符算法。
- 将用户密码发送给验证码接口。
