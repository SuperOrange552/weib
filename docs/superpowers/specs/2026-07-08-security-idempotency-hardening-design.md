# 系统安全限制与幂等性加固设计

日期：2026-07-08

## 目标

统一前后端账号和密码安全规则，为高风险写操作增加前端防重复、Redis 幂等锁和数据库约束，并通过自动化安全测试提高系统在恶意输入、重复点击、接口重放和并发请求下的健壮性。

## 账号与密码规则

### 用户名

- 明文长度 3–32 个字符。
- 允许字母、数字、下划线和中文。
- 注册、普通登录、管理员登录、管理员创建使用同一最大长度。
- 数据库 `users.username` 从 `VARCHAR(50)` 收紧为 `VARCHAR(32)`；迁移前检查是否存在超过 32 个字符的历史账号，存在则停止迁移并输出清单，不静默截断。

### 密码

- 明文长度 8–64 个字符。
- 必须同时包含至少一个大写字母、一个小写字母和一个数字。
- 禁止与用户名或手机号完全相同（忽略用户名大小写）。
- 注册、用户改密、管理员创建、管理员重置密码共用 `CredentialPolicy`。
- 登录仅检查非空和最大 64 位，不在登录时暴露具体密码规则，避免账号枚举。
- BCrypt 哈希长度约 60，数据库 `users.password` 保持 `VARCHAR(100)`，实体字段只校验持久化哈希长度，不使用明文密码规则校验哈希。

### 其他输入边界

- 手机号必须为中国大陆 11 位格式。
- 验证码最多 4 位。
- 管理端理由、备注、职位和公司字段按实体列长度设置前后端 `maxlength` 与 DTO 校验。
- 所有外部字符串先 `trim`，超长数据在进入 Service 和数据库前返回 400 或表单错误。

## 校验架构

新增无状态 `CredentialPolicy`，提供用户名规范化、用户名验证、密码验证以及“密码不得等于用户名/手机号”的组合验证。Controller 负责把错误映射为 HTML 或 JSON，Service 在所有写入口再次校验，防止绕过前端和 Controller。

管理端请求 DTO 使用 Jakarta Validation 注解并在 Controller 添加 `@Valid`。Thymeleaf 表单和 React 管理端同步 `minLength`、`maxLength` 和规则提示，但前端规则仅用于体验，后端始终是权威。

## 幂等架构

采用三层保护：

1. 前端请求期间禁用触发按钮并显示处理中。
2. Redis 幂等拦截器使用 `SET NX EX` 对同一用户、接口、资源和幂等键加锁。
3. 数据库唯一约束或状态条件更新作为最终一致性兜底。

### 幂等键

- HTML 表单使用隐藏字段 `_idempotencyKey`，页面渲染时生成 UUID。
- JavaScript/React 请求使用请求头 `Idempotency-Key`，每次用户主动操作生成 UUID；重试同一操作复用原键。
- 服务端 key：`idempotency:{userId/sessionId}:{method}:{normalizedPath}:{key}`。
- 处理中 TTL 30 秒；成功记录 TTL 10 分钟。
- 同键处理中返回 HTTP 409/业务码 40901“操作处理中，请勿重复提交”。
- 同键成功后再次请求返回业务码 200 和 `duplicate=true`，不重复执行副作用。
- Redis 不可用时记录告警并依赖数据库约束/条件更新，不阻断业务。

### 覆盖操作

- 求职投递：`applications(job_id,user_id)` 唯一约束，重复请求返回已有投递结果。
- 收藏：改为显式“收藏/取消收藏”目标状态，避免 toggle 在重放时反向执行；`favorite_jobs(job_id,user_id)` 唯一约束。
- 公司入驻：同一 Boss 只允许一家公司，`companies.boss_id` 唯一约束。
- 职位发布：表单幂等键防止创建两条职位；编辑按资源 ID 和目标内容执行。
- 面试安排：同一投递重复请求只保留同一目标安排。
- 管理员创建：用户名唯一约束加幂等键。
- 聊天发送：新增 `messages.client_message_id VARCHAR(64)`，并建立 `(sender_id,client_message_id)` 唯一约束；前端重试复用消息 ID。
- 审核、封禁、解封、已读等状态操作使用条件更新或目标状态判断，重复执行返回成功。

## 数据库迁移

新增版本化迁移脚本，执行顺序：

1. 检查超过 32 位的历史用户名。
2. 将 `users.username` 调整为 `VARCHAR(32)`。
3. 确认 `users.password` 为 `VARCHAR(100)`。
4. 确认投递、收藏、公司 Boss 唯一约束存在。
5. 为消息增加 `client_message_id` 和复合唯一索引，允许旧数据为 NULL。
6. 迁移必须可重复执行；DDL 前后查询 information_schema，避免重复索引失败。

不修改或重置现有密码，不删除业务数据。

## 前端交互

新增通用 `submit-guard.js`：表单提交后禁用 submit 按钮，替换为“处理中…”，页面返回或请求失败后恢复。AJAX 操作使用共享 `withIdempotency` 包装器，在 pending 时禁止再次点击，并设置 `Idempotency-Key`。

重点覆盖注册、改密、公司入驻、职位保存、投递、收藏、面试安排、聊天发送和管理后台写按钮。按钮禁用只是体验层，不能替代服务端幂等。

## 错误处理

- 输入不合规返回可读且不泄露敏感信息的错误。
- 登录失败统一返回“账号或密码错误”，锁定状态除外。
- 数据库唯一冲突转换为“操作已完成/记录已存在”，不返回堆栈。
- 幂等处理中返回 409；已成功的重复请求返回成功语义。
- Redis 异常降级时记录结构化日志，不记录密码、验证码、Token 或完整请求体。

## 安全测试

### 凭据测试

- 用户名 2/3/32/33 位边界。
- 非法字符、纯空白和前后空格。
- 密码 7/8/64/65 位边界。
- 缺少大写、小写或数字。
- 密码等于用户名或手机号。
- 注册、改密、管理员创建、重置密码规则一致。
- 登录超长用户名或密码在密码哈希计算前被拒绝。

### 幂等与并发测试

- 相同幂等键并发两次只执行一次。
- 不同幂等键可分别执行。
- 投递、收藏、公司入驻、消息发送的唯一约束兜底。
- 相同聊天 clientMessageId 只保存一条消息。
- 状态操作重复调用结果一致。
- Redis 不可用时数据库兜底仍阻止重复数据。

### 接口安全测试

- 缺失/非法幂等键。
- 越权资源 ID。
- CSRF、限流和幂等拦截器顺序。
- 超长 JSON、表单参数和备注。
- 错误响应不包含密码、哈希、堆栈和内部 SQL。
