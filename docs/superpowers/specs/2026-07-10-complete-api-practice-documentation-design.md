# 微招系统完整接口练习文档设计

## 1. 目标

重新编写一套供学习、练习和手工接口测试使用的微招系统接口文档。读者只阅读文档即可在 Apifox、Postman 或其他 HTTP 工具中完成接口测试，不需要对照前端页面猜测参数、认证方式或调用顺序。

最终同时交付内容一致的两个版本：

- `docs/API_TESTING_COMPLETE.md`
- `docs/微招系统完整接口测试文档.docx`

本次不生成可导入的 OpenAPI、Apifox 或 Postman 集合。

## 2. 事实来源

接口信息必须交叉核对以下来源，不能只复制旧文档：

1. 线上站点 `http://superorange.top` 的实际响应。
2. 线上 `/v3/api-docs` 当前发布的 OpenAPI。
3. 项目中全部 Controller、DTO、拦截器、安全配置和实体字段。
4. 登录页实际生成的 CSRF 字段、Session Cookie 和验证码响应头。
5. 现有 `docs/API_TESTING.md`，仅作为历史参考，不作为最终事实来源。

若源码、旧文档与线上行为冲突，以当前部署版本的实际行为为准，并在文档中明确说明差异。

## 3. 收录范围

主文档收录所有可用于数据查询、业务操作或身份认证测试的接口：

- 公共认证：验证码、普通用户登录、注册、退出、修改密码、账号占用检查。
- 求职者：职位列表、职位详情、公司详情、投递、收藏、撤回、简历、通知、会话。
- Boss：企业入驻与维护、职位保存/删除/重开、申请处理、面试安排、简历查看、统计。
- 聊天：文件上传与下载、消息发送、消息同步、在线状态、已读。
- 管理后台：登录、当前管理员、用户、公司、职位、子管理员、审计日志、仪表盘、导出。
- 地图/地理编码等独立数据接口。

仅返回 HTML 页面且不承担业务提交的 GET 路由不混入主接口清单，统一放到“页面型路由附录”。

## 4. 文档组织

文档按照真实练习顺序编排：

1. 文档说明、系统地址、工具设置。
2. 通用响应结构与认证概念。
3. 普通用户完整登录流程。
4. 管理员完整登录流程。
5. 注册、退出、修改密码等账号接口。
6. 求职者接口。
7. Boss 和企业接口。
8. 职位、投递和收藏接口。
9. 简历接口。
10. 聊天、文件和通知接口。
11. 管理后台接口。
12. 地图等公共数据接口。
13. 页面型路由附录。
14. 状态码、业务错误、限流和幂等规则。
15. 正常、异常、边界、权限和重复提交练习清单。
16. 推荐练习顺序与变量记录表。

普通用户登录和管理员登录必须出现在目录前部，并明确标识：

- 普通用户：`POST /login`
- 管理员：`POST /api/admin/auth/login`

## 5. 单个接口的固定模板

每个接口必须包含以下内容：

1. 接口名称和用途。
2. 请求方法、相对路径和完整线上 URL。
3. 适用角色。
4. 前置条件。
5. 认证要求：Session Cookie、JWT、CSRF、验证码和幂等键分别说明。
6. `Content-Type`。
7. Header 参数表。
8. Path 参数表。
9. Query 参数表。
10. Form 或 JSON Body 参数表。
11. 每个参数的名称、类型、是否必填、规则、示例和说明。
12. 可直接复制的 cURL 或 HTTP 请求示例。
13. 成功响应示例。
14. 常见失败响应或跳转行为。
15. 正常、异常、边界、权限和重复请求测试点。
16. 后续可衔接测试的接口。

没有某一类参数时不放空表，直接标注“无”。Map 类型请求体必须根据 Controller 中实际读取的字段展开，不能只写 `object`。

## 6. 普通用户登录设计

普通用户登录是 Session + CSRF + 图片验证码组合流程，文档必须给出脱离浏览器页面的完整步骤：

1. `GET /login`：创建并保存 `JSESSIONID`，从 HTML 中提取隐藏字段 `_csrf`。
2. `GET /captcha`：复用同一个 Cookie，查看 PNG 响应并记录验证码；说明 `X-Captcha-Expires-In: 120`。
3. `POST /login`：以 `application/x-www-form-urlencoded` 提交 `username`、`password`、`captcha`、`_csrf`。
4. 登录成功：说明 302 重定向、Session 更新以及 `remember_token`/`jwt_token` Cookie。
5. 登录失败：说明可能返回登录页 HTML，而不是统一 JSON；列出验证码、账号密码、锁定、CSRF和限流场景。

文档必须提醒测试工具启用 Cookie Jar，并保证三个请求使用同一个会话。验证码需要人工读取图片，不能在文档中伪造固定值。

## 7. 管理员登录设计

管理员登录作为标准 JSON API 单独说明：

1. `POST /api/admin/auth/login` 提交 `username` 和 `password`。
2. 从 `data.token` 获取 JWT。
3. 后续请求携带 `Authorization: Bearer <token>`。
4. 按 `super_admin`、`auditor`、`viewer` 说明权限差异。
5. 登录失败、账号锁定、非管理员账号和无效 Token 的响应示例必须分别说明。

不得在文档中写入真实生产密码或服务器凭据，账号和密码示例使用明显的练习占位值。

## 8. 通用规则

文档集中说明并在相关接口重复标注以下规则：

- 用户名：新建时 3–32 位，允许字母、数字、下划线和中文。
- 密码：新建/修改/重置时 8–64 位，包含大写字母、小写字母和数字。
- 手机号：11 位中国大陆手机号格式。
- 验证码：4 位、120 秒有效、刷新冷却和频率限制。
- `Idempotency-Key`：8–128 位，字符范围为字母、数字、点、下划线、冒号和连字符。
- `clientMessageId`：聊天消息的客户端唯一标识。
- 分页：明确页面从 0 还是 1 开始、默认值和示例。
- 日期时间：明确 ISO-8601 格式示例。
- 混淆 ID：`encodedId` 不能直接用数据库数字 ID 替代，并说明从哪个前置响应取得。

## 9. 响应与错误说明

文档区分三类响应：

1. JSON API 的 `Result<T>`：`code`、`msg`、`data`。
2. 幂等拦截器响应：`code`、`message`、`duplicate`。
3. HTML 表单接口：200 页面或 302 重定向。

状态码至少覆盖：200、302、400、401、403、404、409、429、500。每个状态码都说明常见触发原因及下一步排查方式。

## 10. Word 与 Markdown 版式

Markdown 版本强调可搜索和可复制：

- 稳定目录和标题层级。
- 参数使用统一表格。
- 请求与响应放在带语言标识的代码块中。
- 重要前置条件使用醒目提示。

Word 版本强调阅读和打印：

- 封面、版本、生成日期和适用环境。
- 自动目录或可更新目录域。
- 统一标题、正文、表格和代码样式。
- 页眉显示“微招系统完整接口测试文档”，页脚显示页码。
- 表格不越界，接口标题避免落在页末，代码块保持可读。

两个版本的接口数量、字段、示例和章节顺序必须一致。

## 11. 验证标准

交付前执行以下验证：

1. 从源码生成 Controller 映射清单，与文档逐项比对。
2. 与线上 `/v3/api-docs` 的当前路径清单比对。
3. 单独确认 `/login`、`/register` 等未完整出现在旧 OpenAPI 中的接口已补入文档。
4. 搜索所有 `@RequestParam`、`@PathVariable` 和 `@RequestBody`，确保请求字段无遗漏。
5. 检查所有 `@Idempotent` 接口均标明幂等键。
6. 检查所有管理员接口均标明 JWT 与角色要求。
7. 使用线上站点实际执行匿名安全请求，验证登录页、验证码、公开查询和错误响应。
8. Markdown 执行链接、代码块和结构检查。
9. Word 生成后渲染为页面图片或 PDF，逐页检查目录、表格、分页、中文字体和代码块。
10. 对比 Word 与 Markdown 的接口标题清单，保证完全一致。

不使用会修改生产业务数据的请求进行线上验证；写接口示例以源码契约和隔离测试为准。

## 12. 发布方式

最终文件保存在本地项目 `docs` 目录，并在交付回复中提供绝对路径。若服务器连接可用，同时同步到 `/opt/weib/source/docs/` 供后续取用，但不因为文档更新而重启业务服务。
