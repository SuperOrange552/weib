# Redis 缓存与投诉审核处罚设计

**日期：** 2026-07-11

## 1. 背景

当前项目已经接入 Spring Data Redis 和 Spring Cache，但职位、公司、简历和用户资料的业务读取仍主要由 JPA 直接访问 MySQL。随着访问量增加，需要把 Redis 调整为公开读请求的第一读取层，降低数据库压力，同时保证缓存击穿、缓存穿透、缓存雪崩和并发写入导致的数据不一致可控。

项目还需要在 Boss 端和求职者端提供投诉入口。投诉对象覆盖用户账号、具体职位、公司资料/照片、简历及简历头像/照片。后台审核后可以驳回投诉、下架内容、禁言、禁止发布或封禁账号，并支持临时处罚和永久处罚。

## 2. 目标与非目标

### 2.1 目标

1. 对 Boss、求职者、公司、职位、简历的公开读取采用 Cache-Aside：先查 Redis，未命中时回源 MySQL 并回填。
2. 对热点 Key 提供加载锁、空值缓存、TTL 抖动、限流和受控降级，避免 Redis 或数据库被高并发拖垮。
3. 在数据库写入成功后删除相关缓存，并执行延迟二次删除，降低并发读写产生旧值回填的概率。
4. 增加统一投诉记录、证据、审核、处罚、通知和审计链路。
5. 处罚结果在登录、聊天、职位发布/编辑、简历保存/发布和公开查询链路中生效。
6. 兼容已有旧数据：头像、图片、审核字段为空时不阻断读取和业务操作。

### 2.2 非目标

1. 本阶段不把 Redis 作为唯一数据源，不移除 MySQL 持久化。
2. 本阶段不引入 Kafka、RabbitMQ 等消息队列。
3. 本阶段不自动生成虚假头像或虚假业务数据；旧数据补齐作为后续独立数据治理任务。
4. 本阶段不删除历史投诉、处罚和审计记录。

## 3. 总体架构

采用统一 Cache-Aside 服务组件，业务 Service 不直接散落调用 Redis。读取流程如下：

```text
HTTP/WebSocket 请求
        |
        v
统一读取服务查询 Redis
   | 命中                 | 未命中
   v                      v
返回展示 DTO        获取 SET NX EX 加载锁
                         |
                         v
                   二次检查 Redis
                    |        |
                  命中      仍未命中
                    |        v
                    |   查询 MySQL
                    |        |
                    |   写入 Redis
                    |        |
                    +----> 返回 DTO
```

写入流程：

```text
开启数据库事务
  -> 更新 MySQL
  -> 事务提交成功
  -> 删除相关 Redis Key
  -> 延迟 800ms 再删除一次
```

缓存只保存公开展示所需字段或安全 DTO，不缓存密码、记住登录 Token、验证码答案、CSRF Token 和完整认证对象。

## 4. Redis 缓存策略

### 4.1 Key 规范

| 数据 | Key 示例 | 默认 TTL |
|---|---|---:|
| 用户公开资料 | `cache:user:public:{userId}` | 10 分钟 |
| 公司详情 | `cache:company:{companyId}` | 30 分钟 |
| 职位详情 | `cache:job:{jobId}` | 5 分钟 |
| 职位列表 | `cache:jobs:list:{queryHash}:{page}:{size}` | 2-5 分钟 |
| 简历公开详情 | `cache:resume:public:{resumeId}` | 10 分钟 |
| 简历按用户查询 | `cache:resume:user:{userId}` | 10 分钟 |
| 空值占位 | 对应业务 Key | 30 秒 |
| 回源加载锁 | `lock:cache:load:{namespace}:{idOrHash}` | 5-10 秒 |
| Redis 限流 | `rate:redis-read:{route}:{clientKey}` | 按接口窗口 |

列表 Key 必须包含完整查询条件、页码和页大小的稳定 Hash，禁止把未经规范化的长查询字符串直接作为 Key。

### 4.2 击穿、穿透和雪崩

- **击穿：** 对同一详情 Key 使用 `SET NX EX` 加载锁；未获得锁的请求短暂退避后重读 Redis，超时后才受控回源。
- **穿透：** MySQL 确认不存在的 ID 写入短 TTL 空值对象；空值响应不得泄露数据库内部异常。
- **雪崩：** 正常 TTL 加入小范围随机抖动；不在整点批量刷新所有列表缓存。
- **大 Key：** 列表必须分页，单页大小服务端限制为 100；禁止把完整 JPA PageImpl 直接序列化到 Redis。

### 4.3 一致性和延迟双删

1. 所有新增、编辑、下架、审核、封禁、解封和处罚撤销操作必须先完成数据库事务。
2. 事务提交成功后删除详情 Key、关联用户 Key、列表 Key 和受影响的公司/职位/简历 Key。
3. 使用受控异步任务在约 800ms 后再次删除同一组 Key，避免并发读在第一次删除和事务提交之间把旧值回填。
4. 延迟删除失败时记录告警并保留重试信息；不能回滚已经提交的数据库事务。
5. 读请求只接受带版本/更新时间的安全 DTO；需要强一致的管理员详情页在缓存异常时直接回源数据库。

### 4.4 Redis 限流和降级

- 公开读接口按 IP、用户和路由设置窗口限流；登录用户优先使用用户维度，匿名请求使用 IP 维度。
- 同一 Key 的并发回源请求受加载锁保护；超过等待时间返回缓存暂不可用或受控数据库结果，不进行无限重试。
- Redis 连接异常时允许短时降级到 MySQL，但使用数据库查询超时、分页上限和告警日志保护数据库。
- Redis 恢复后新请求自动重新填充缓存；不在应用内长期保存大批量本地缓存。

## 5. 投诉领域模型

### 5.1 `complaints` 投诉记录

| 字段 | 说明 |
|---|---|
| `id` | 投诉 ID |
| `reporter_id` | 投诉人用户 ID |
| `target_type` | `USER`、`JOB`、`COMPANY`、`RESUME`、`MEDIA` |
| `target_id` | 对象 ID；媒体对象使用所属资源 ID |
| `category` | 虚假职位、虚假照片、诈骗、骚扰、垃圾信息、其他 |
| `description` | 投诉说明，限制长度并做 HTML 清理 |
| `evidence_urls` | 最多 5 个证据文件地址，使用受控 JSON/文本格式 |
| `status` | `PENDING`、`RESOLVED`、`REJECTED` |
| `reviewer_id` | 审核管理员 ID |
| `review_reason` | 审核结论和处理说明 |
| `reviewed_at` | 审核时间 |
| `created_at` / `updated_at` | 创建和更新时间 |

约束：

- 同一投诉人对同一对象只能存在一条 `PENDING` 投诉；
- 投诉人不能投诉自己或自己拥有的对象；
- 投诉说明有最大长度；证据限制类型、数量和大小；
- 投诉对象不存在时返回业务错误，不创建悬空投诉；
- 投诉完成后仍保留完整记录。

### 5.2 `user_sanctions` 处罚记录

| 字段 | 说明 |
|---|---|
| `id` | 处罚 ID |
| `user_id` | 被处罚用户 |
| `sanction_type` | `MUTE`、`PUBLISH_BAN`、`ACCOUNT_BAN` |
| `target_type` / `target_id` | 关联的内容对象，可为空 |
| `source_complaint_id` | 来源投诉，可为空（后台直接处罚时允许为空） |
| `reason` | 处罚原因 |
| `starts_at` | 生效时间 |
| `ends_at` | 到期时间，`NULL` 表示永久 |
| `status` | `ACTIVE`、`EXPIRED`、`REVOKED` |
| `admin_id` | 操作管理员 |
| `created_at` / `updated_at` | 创建和更新时间 |

处罚类型含义：

- `MUTE`：禁止发送文本和文件消息；
- `PUBLISH_BAN`：Boss 不能新增/编辑职位，求职者不能保存/发布简历；
- `ACCOUNT_BAN`：禁止登录和已登录业务操作；
- 具体职位、公司、简历或媒体下架通过其业务表状态/可见性字段执行，并关联处罚记录。

## 6. 用户端投诉流程

Boss 和求职者在用户卡片、职位详情、公司资料、简历预览和图片查看区域提供“投诉”入口。

1. 前端带入 `targetType` 和 `targetId`；
2. 用户选择投诉分类，填写说明，可上传最多 5 个证据；
3. 后端校验登录状态、对象存在性、归属关系和重复投诉；
4. 创建 `PENDING` 投诉并返回投诉编号；
5. 后台审核结束后，投诉人可查看处理状态和审核意见；
6. 处理结果通过站内通知发送给相关用户；敏感审核信息只展示必要内容。

## 7. 管理后台流程与接口

沿用现有 `/api/admin` 前缀、JWT 管理员认证、RBAC 和 `Result` 响应格式，新增投诉审核模块：

| 方法 | 路径 | 作用 |
|---|---|---|
| `GET` | `/api/admin/complaints` | 分页筛选投诉 |
| `GET` | `/api/admin/complaints/{id}` | 查看投诉、证据和对象快照 |
| `POST` | `/api/admin/complaints/{id}/reject` | 驳回投诉 |
| `POST` | `/api/admin/complaints/{id}/resolve` | 通过投诉并执行内容/用户处理 |
| `POST` | `/api/admin/sanctions` | 创建临时或永久处罚 |
| `POST` | `/api/admin/sanctions/{id}/revoke` | 撤销未到期处罚 |
| `GET` | `/api/admin/sanctions` | 查询处罚历史和当前状态 |

处理请求必须携带处理理由、处罚类型和可选 `endsAt`。所有写操作使用现有幂等机制和权限检查：审核员可以处理投诉；账号封禁、永久处罚和撤销处罚需要超级管理员权限。

## 8. 处罚生效点

- 登录：查询用户当前有效 `ACCOUNT_BAN`，拒绝登录并返回通用提示；
- 已登录请求：统一用户状态/处罚检查，避免只限制登录而绕过业务接口；
- 聊天发送 REST 和 WebSocket：发送前检查 `MUTE`；
- Boss 职位创建、编辑、重新发布：检查 `PUBLISH_BAN`；
- 求职者简历保存、发布和附件更新：检查 `PUBLISH_BAN`；
- 公开职位、公司、简历和媒体读取：过滤已下架或不可见对象；
- 处罚到期：读取时判断 `ends_at`，自动视为失效并异步更新状态；
- 处罚、解封、撤销和内容下架后清理相关 Redis Key。

## 9. 数据库迁移与旧数据兼容

1. 新增幂等迁移脚本创建投诉表、处罚表、索引和必要的内容可见性/审核字段。
2. 迁移前备份数据库，并在测试库执行完整迁移和回滚演练。
3. `avatar`、图片地址、审核状态等旧字段允许为空；展示层使用默认头像或“未上传”状态，不把空值当作异常。
4. 不在本次功能中批量制造业务数据；如果后续需要演示数据，单独编写可重复执行的 seed 脚本并标记来源。
5. 数据库约束、实体、DTO 和 Redis DTO 必须保持字段语义一致，避免缓存返回与数据库返回结构不一致。

## 11. 回滚方案

1. 关闭缓存优先读取开关，应用临时回到受控 MySQL 读取；
2. 保留新增投诉和处罚表，不删除历史数据；
3. 若投诉模块异常，关闭前端入口和写入接口，但保留后台审计数据；
4. 若处罚逻辑异常，优先停止新增处罚，人工核对有效处罚记录后再恢复；
5. 任何代码回滚都不回滚已提交的投诉、处罚和审计记录，避免合规链路断裂。

## 12. 已确认的设计决策

- Redis 是缓存，不是主存储；
- MySQL 是最终持久化来源；
- 采用 Cache-Aside、加载锁、空值缓存、TTL 抖动和延迟双删；
- 投诉覆盖账号、职位、公司资料/照片、简历及简历照片；
- 处罚支持临时和永久；
- 旧数据缺少头像等字段时兼容读取，不在本阶段强制造数据；
- 不引入新的消息队列，先使用现有 Spring Boot、Redis、MySQL、审计日志和通知能力。
