# 微招 Android App 功能补全与分页性能优化设计

## 目标

将 Android App 补全为可供求职者和招聘者实际使用的移动端：职位及人才列表采用服务端分页和 Redis 缓存；完成双身份登录、投递与招聘流程、简历授权、实时消息与强提醒、论坛、投诉及申诉，并与现有 Web、管理员后台和生产服务器保持接口及权限一致。

## 范围与边界

- Android 仅包含求职者端和招聘者端，不包含管理员端。
- 同一账号可开通一个或两个身份；每次登录必须选择身份，切换身份必须退出重新登录。
- 消息、会话和业务数据按账号与当前身份隔离。
- 手机端同一账号只允许一个有效会话，电脑端同理；手机和电脑可同时在线。
- 管理员审核仍由 Web 管理后台完成，App 只提供投诉、申诉和结果查看入口。

## 方案选择

采用“服务端分页 + App 懒加载 + Redis 分页缓存”。列表接口返回轻量摘要，详情按 ID 获取；搜索条件在服务端执行；App 首屏加载 20 条，接近底部自动加载下一页。相比一次返回全部数据，此方案降低网络、解析、内存和 Compose 重组成本，并能在数据增长后保持稳定。

暂不引入 Room 离线数据库。本轮只保留当前页面状态和必要的会话数据，避免增加离线同步与冲突处理复杂度。

## 架构

### 服务端

- 列表接口统一接受 `page`、`size`、查询条件和排序参数；`page` 从 0 开始，`size` 默认 20、最大 50。
- 返回 `content`、`page`、`size`、`totalElements`、`totalPages`、`first`、`last`。
- 职位、人才、投递、会话、论坛帖子、评论、投诉和申诉均使用分页查询。
- 对高频只读列表使用 Redis Cache-Aside。缓存键包含身份、过滤条件摘要、页码和排序；设置有抖动的 TTL，避免同一时间集中失效。
- 写操作提交数据库后删除对应实体缓存和列表命名空间版本；读取时缓存未命中才访问数据库，避免缓存脏数据长期存在。
- 使用现有幂等键、限流、身份权限和会话机制保护写接口。

### Android

- Retrofit API 显式接收分页和查询参数。
- 每类分页列表使用独立的分页状态：数据、页码、是否首屏加载、是否追加加载、是否还有下一页及错误。
- 搜索条件变更采用短延迟防抖，并取消旧查询结果写回。
- LazyColumn 接近末尾时请求下一页；同一页请求只允许一个在途任务。
- 下拉刷新从第 0 页重新加载；追加失败只重试当前页，不清空已加载数据。
- 列表使用稳定 ID，合并下一页时按 ID 去重。

## 功能设计

### 登录与会话

- 登录页选择求职者或招聘者身份。
- 服务端验证账号是否拥有所选身份；未开通招聘者身份时返回明确错误。
- 401 会话失效、异地登录、密码修改和账号封禁均清除本地 Token，停止实时连接并返回登录页。
- 异地登录显示安全提醒；普通过期显示重新登录提醒。

### 求职者端

- 分页浏览与搜索职位，查看详情、收藏和投递。
- 查看投递记录、状态、更新时间并撤回允许撤回的投递。
- 编辑简历，上传本地头像及附件。
- 收到完整简历请求后可同意或拒绝；同意后仅授权对应招聘者查看。
- 使用消息、论坛、投诉、申诉和个人中心。

### 招聘者端

- 查看招聘数据概览。
- 创建、编辑、关闭和重新开放职位。
- 分页搜索公开的部分在线简历，只展示允许公开的摘要字段。
- 发起私聊及完整简历授权请求。
- 查看投递并更新流程状态。
- 使用消息、论坛、投诉、申诉和企业资料。

### 消息与强提醒

- 前台实时通道接收消息和安全事件。
- 后台 WorkManager 定时补偿拉取通知事件。
- 私聊、投递被查看、投递状态变更、完整简历请求、投诉和申诉结果触发系统通知。
- 使用服务端事件 ID 本地去重，实时与轮询不会重复提醒。
- 通知点击进入对应业务详情；若会话已失效则先回登录页。

### 论坛

- 板块和标签筛选、关键词搜索、分页帖子列表及详情。
- 从系统文件选择器选择本地图片后上传。
- 支持发布图文、评论、点赞、收藏；显示作者头像、发布时间及统计数。
- 删除和编辑只能由作者或有权限的管理员执行。

### 投诉与申诉

- 职位、公司、用户、帖子和评论均可发起投诉，选择原因并提交文字及图片证据。
- 用户可查看投诉处理状态和结果。
- 被处罚账号可提交申诉及证据并查看审核结果。
- 禁言、禁止发布和账号封禁由服务端统一拦截，不能只依赖前端按钮状态。

## 错误处理

- 401：清理会话并回登录页。
- 403：展示身份未开通或权限不足，不自动重试。
- 409：展示重复投递、重复操作或状态冲突。
- 429：展示限流剩余等待时间，禁止立即连续重试。
- 5xx/网络异常：保留已加载内容，提供局部重试。
- 上传失败：保留表单和已选文件，允许重新上传。

## 性能与一致性

- 默认每页 20，最大 50；列表 DTO 不携带大文本和二进制内容。
- 搜索请求防抖、旧请求取消、页请求互斥、数据按 ID 去重。
- Redis 采用 Cache-Aside、短 TTL 抖动、空值短缓存和写后失效；防止缓存穿透和雪崩。
- 数据库是最终真相；缓存不可用时降级读取数据库，但限流仍生效。
- 图片使用服务端缩略图地址，避免列表加载原图。

## 测试与验收

- 后端覆盖分页边界、最大 size、权限、缓存命中与失效、重复写幂等、会话互踢及处罚拦截。
- Android 覆盖分页状态、去重、失败重试、身份导航、401 退出和通知事件去重。
- 完成求职者与招聘者核心流程联调。
- 运行后端测试、`testProdDebugUnitTest` 和 `assembleProdDebug`。
- 将新 APK 安装到模拟器，验证首屏仅请求 20 条且滚动才加载下一页。
- 最后提交分支、合并主分支、部署服务器并执行运行时健康检查。

## 实施顺序

1. 修复会话过期闭环并建立分页状态基础设施。
2. 完成职位分页、搜索、详情、收藏和投递。
3. 完成招聘者职位、人才搜索、投递处理和简历授权。
4. 完成消息、实时通知和系统强提醒。
5. 完成论坛图文、互动与搜索。
6. 完成投诉、申诉及管理员权限联动。
7. 全量测试、模拟器安装、生产部署和主分支合并。
