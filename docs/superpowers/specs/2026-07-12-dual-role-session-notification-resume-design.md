# 微招双身份、单端会话、强提醒与在线简历设计

**日期：** 2026-07-12  
**状态：** 已确认  
**适用端：** Web 求职者端、Web Boss 端、Android 求职者端、Android Boss 端；管理员仍仅使用 Web 管理后台。

## 1. 目标与范围

本次升级同时解决五类需求：

1. 补齐第一版 Android App 尚未完成的功能，使求职者端和 Boss 端形成可实际使用的完整业务闭环。
2. 一个账号可开通一个或多个身份，登录时必须选择求职者或招聘者身份。
3. 同一账号允许 Web 与 Mobile 同时在线，但每种终端最多保留一个在线会话；同端新登录会踢掉旧会话。
4. App 对私聊、简历查看、投递状态和安全事件提供前台、后台和进程被杀后的分层强提醒。
5. 通过企业审核的 Boss 可搜索公开的脱敏在线简历，私聊求职者并请求完整简历；完整资料必须经求职者明确授权。

管理员登录、管理员权限和管理后台页面不进入 Android App，但 Web 管理后台必须同步增加双身份授权、企业审核、会话管理、通知审计和简历授权审计能力。

## 2. 已确认的业务规则

### 2.1 身份开通

- `seeker_ahua` 同时开通 `SEEKER` 和 `BOSS`。
- 现有 `boss_zhang`、`boss_li`、`boss_wang`、`boss_zhao`、`boss_chen` 仅保留 `BOSS`。
- 其他现有求职者仅保留 `SEEKER`。
- 新注册账号默认只开通 `SEEKER`。
- 未开通 `BOSS` 的账号选择招聘者登录时返回 `403 ROLE_NOT_ENABLED`，并显示企业权限申请入口。
- 企业权限开通不等同于企业审核通过；未完成企业审核时可以进入受限 Boss 工作台，但不能发布职位、搜索在线简历或发起简历请求。

### 2.2 身份切换与隔离

- 登录请求必须包含 `selectedRole`。
- 本次会话固定使用所选身份；切换身份必须退出并重新完成账号、密码和验证码登录。
- 同一账号的求职者与 Boss 被视为两个独立业务个体。
- 消息、会话、通知、收藏、投递、求职简历、公司、职位、投诉和申诉均按 `userId + activeRole` 进行权限隔离。
- 身份展示资料独立保存，聊天和论坛显示当前身份对应的昵称与头像。

### 2.3 单端会话

- 同一账号最多有一个 `WEB` 会话和一个 `MOBILE` 会话，角色不参与槽位划分。
- 同一账号可以同时保留一个电脑端和一个手机端会话。
- 新手机登录会踢掉该账号原手机会话；新电脑登录会踢掉原电脑会话。
- 旧端通过 WebSocket 立即收到 `FORCE_LOGOUT`；若实时事件未到达，则下一次请求通过会话版本校验返回 `401 KICKED`。
- 旧端必须清除本地 Token/Session，跳转登录页并提示：
  `你的账号已在其他设备登录，请检查密码是否泄露；如有风险，请立即修改密码。`
- 修改密码、账号封禁或管理员强制下线时，清除该账号的 `WEB` 和 `MOBILE` 两个会话槽位。

## 3. 数据模型

### 3.1 账号与身份

新增 `user_roles`：

- `id`
- `user_id`
- `role_type`：`SEEKER` / `BOSS`
- `status`：`ACTIVE` / `PENDING` / `DISABLED`
- `enabled_at`
- `enabled_by`
- `created_at` / `updated_at`
- 唯一约束：`(user_id, role_type)`

新增 `role_profiles`：

- `id`
- `user_id`
- `role_type`
- `nickname`
- `avatar`
- `bio`
- `created_at` / `updated_at`
- 唯一约束：`(user_id, role_type)`

`users.role` 在迁移期保留，只用于旧客户端兼容和回滚，不作为新接口的最终授权依据。

### 3.2 会话注册

Redis 键：

- `login:slot:{userId}:WEB`
- `login:slot:{userId}:MOBILE`

值包含：`sessionTokenId`、`activeRole`、`issuedAt`、`lastSeenAt`、`deviceIdHash`、`ip`、`userAgent`。TTL 与访问令牌生命周期一致，活跃请求可按上限续期。

JWT 新增：`sid`、`activeRole`、`clientType`。每个受保护请求都校验 JWT 的 `sid` 是否仍等于 Redis 当前槽位，防止旧 Token 在被踢后继续使用。

### 3.3 通知事件

新增 `notification_events`：

- `event_id`：全局唯一，用于去重
- `recipient_user_id`
- `recipient_role`
- `event_type`
- `priority`：`NORMAL` / `HIGH` / `URGENT`
- `title`
- `content`
- `payload_json`
- `read_at`
- `created_at`

新增 `mobile_push_tokens`：按用户、终端、应用安装实例保存 FCM Token、状态和最后活跃时间。强制下线后可保留设备推送 Token，但敏感通知发送前仍校验当前会话与身份。

### 3.4 在线简历授权

在简历上增加：

- `visibility`：`PRIVATE` / `PUBLIC_SUMMARY`
- `expected_job_title`
- `expected_city`
- `work_years`
- `summary`

新增：

- `resume_access_requests`：请求者 Boss 身份、公司、关联职位、说明、状态和处理时间。
- `resume_access_grants`：指定求职者身份向指定 Boss 身份授予完整简历访问权，记录简历版本、授权时间和撤回时间。
- `resume_view_logs`：记录完整简历查看人、身份、公司、时间和请求来源。

## 4. 登录与权限流程

### 4.1 登录页

Web 和 Android 登录页增加求职者/招聘者选择器。默认选择求职者，但仅作为界面默认值，后端仍验证该账号是否开通对应角色。

登录请求字段：

- `username`
- `password`
- `captcha`
- `selectedRole`
- `clientType`：Web 由服务端确定为 `WEB`，App 固定为 `MOBILE`，客户端不能伪造切换。
- `deviceId`：App 安装实例 ID 或 Web 浏览器实例 ID，只存哈希。

登录成功后：

1. 验证验证码、凭据、账号状态和角色开通状态。
2. 生成新的 `sid`。
3. 原子覆盖 Redis 对应终端槽位。
4. 向旧 `sid` 发送 `FORCE_LOGOUT`。
5. 返回包含 `activeRole` 和 `clientType` 的新 Token/Session。

### 4.2 统一错误契约

- `401 EXPIRED`：会话自然过期。
- `401 KICKED`：同端被新登录挤掉。
- `401 PASSWORD_CHANGED`：密码变更导致会话失效。
- `403 ROLE_NOT_ENABLED`：账号未开通所选身份。
- `403 COMPANY_NOT_APPROVED`：Boss 身份已开通但企业未审核通过。
- `403 ACCOUNT_BANNED`：账号封禁，仅允许访问申诉路径。

Web 对上述错误跳转登录页并显示具体原因；App 使用全局拦截器弹窗、清理状态并导航登录页。

## 5. 消息与强提醒

### 5.1 事件来源

- 私聊新消息。
- Boss 查看投递简历。
- 投递状态变化。
- 面试邀请。
- 完整简历请求、同意、拒绝或撤回。
- 投诉和申诉结果。
- 账号同端新登录、修改密码、账号封禁。

### 5.2 投递与送达

业务事务先写 `notification_events`，提交后再发布实时事件：

1. App 前台：WebSocket 弹窗、声音和震动。
2. App 后台：Android 通知栏，点击进入对应职位、聊天、投递或申诉页面。
3. App 进程被杀：FCM 推送。
4. FCM 未配置或失败：WorkManager 定期拉取未读事件作为补偿。

每种通道使用同一个 `eventId`，App 本地维护已处理事件集合，避免重复弹窗。`URGENT` 强制下线事件优先于普通业务通知。

FCM 无法在缺少 Firebase 项目和 `google-services.json` 时完成真实远程推送；在配置到位前，前台 WebSocket 和后台 WorkManager 保持可用，代码预留正式 FCM 接入点。

## 6. 在线简历与授权流程

### 6.1 公开摘要

只有求职者主动设置 `PUBLIC_SUMMARY` 的简历进入搜索索引。审核通过的 Boss 可按职位、技能、城市、学历和工作年限搜索。

公开字段：头像、昵称、学历、学校、专业、技能标签、工作年限、精简自我介绍、期望职位、期望城市。

隐藏字段：真实姓名、手机号、邮箱、出生日期、完整工作经历、完整项目经历、完整自我介绍和简历附件。

### 6.2 请求与授权

1. Boss 在摘要卡片发起私聊。
2. Boss 发送完整简历请求，必须选择关联职位并填写请求说明。
3. 求职者收到高优先级提醒并选择同意或拒绝。
4. 同意后创建只针对该 Boss 身份的授权记录，并在聊天中发送完整简历卡片。
5. 求职者可撤回后续访问；已在授权期间合法查看或下载的内容无法从对方设备远程删除。
6. 所有完整简历查看和下载写入审计日志。

搜索、私聊和请求接口按 Boss、公司和 IP 限流；批量抓取、短时间大量私聊或请求会被阻断并记录。

## 8. 兼容迁移

迁移脚本必须幂等执行：

1. 备份生产数据库和当前 JAR。
2. 新建表和索引，不删除旧字段。
3. `seeker_ahua` 写入 `SEEKER` 与 `BOSS`。
4. 现有 `boss_*` 写入 `BOSS`。
5. 其余用户写入 `SEEKER`。
6. 为每个角色创建初始 `role_profiles`，复制当前昵称和头像。
7. 在兼容期内，新授权逻辑优先读 `user_roles`；不存在迁移数据时才回退旧 `users.role`。
8. 生产验证稳定后再停止旧字段回退，但本次不删除 `users.role`。

## 9. Web 管理后台权限升级

管理后台沿用现有 `super_admin`、`auditor`、`viewer` 三级权限，并新增细粒度能力校验；前端隐藏按钮不能代替后端授权。

### 9.1 超级管理员 `super_admin`

- 查看账号已经开通的全部身份及状态。
- 为账号开通、禁用或恢复 `SEEKER` / `BOSS` 身份。
- 不允许删除账号最后一个有效身份；需要停用整个账号时使用账号封禁。
- 查看账号的 Web/Mobile 在线会话、登录时间、IP、设备摘要和当前身份。
- 强制下线指定终端或全部终端；修改密码和账号封禁自动触发全部终端下线。
- 查看通知事件投递状态、失败原因并对失败事件执行受控重试。
- 查看完整简历授权、撤回和查看日志。
- 所有角色和会话操作必须写入 `audit_logs`，记录操作者、目标账号、原值、新值和原因。

### 9.2 审核员 `auditor`

- 审核企业入驻及 Boss 权限申请，通过后开通或激活 `BOSS` 身份。
- 审核投诉和申诉，并按已有权限执行禁言、禁止发布或账号封禁。
- 查看与案件相关的身份资料、通知记录、简历请求及查看日志。
- 可以撤销违规的完整简历授权，但不能直接修改超级管理员或其他管理员权限。
- 不能任意开通求职者身份、不能修改密码、不能执行无案件依据的批量强制下线。

### 9.3 只读管理员 `viewer`

- 只读查看用户、身份、公司、职位、投递、投诉申诉、在线会话摘要和通知统计。
- 手机号、邮箱、设备标识和 IP 默认脱敏。
- 不可审批、开通身份、修改资料、重试通知、撤回授权或强制下线。

### 9.4 后台页面与接口

用户详情页新增：

- “身份与权限”：展示求职者/Boss 开通状态、企业审核状态及操作历史。
- “在线会话”：分别展示 Web 和 Mobile，会话状态实时刷新，支持授权用户强制下线。
- “通知记录”：按身份、类型、优先级、已读和投递状态筛选。
- “简历授权审计”：展示请求、同意、拒绝、撤回和完整简历查看记录。

企业审核页面在通过企业入驻时必须原子完成公司审核和 `BOSS` 身份激活；驳回时保留申请记录并将身份保持为 `PENDING` 或 `DISABLED`，不能产生可发布职位的半成功状态。

新增管理接口必须复用现有管理员 JWT 与 RBAC 体系，并为角色变更、会话强退、通知重试和授权撤销添加幂等键与审计日志。

## 10. 测试与验收

### 10.1 后端

- 角色迁移准确性与幂等性。
- 未开通身份、企业未审核、账号封禁权限矩阵。
- 同账号 Web/Mobile 并存及同端互踢。
- 被踢 Token 无法继续访问。
- 消息按 `userId + activeRole` 隔离。
- 事件持久化、去重和补偿。
- 简历公开脱敏、授权、拒绝、撤回、查看审计和水平越权。

### 10.2 Android

- 求职者与 Boss 两套完整导航及功能操作。
- HTTP 生产环境和 HTTPS 本地环境。
- 前台、后台通知和 WorkManager 补偿。
- 401/403 全局处理和强制下线弹窗。
- 文件和图片使用系统选择器上传并显示进度。
- 单元测试、Lint、production APK 构建、ADB 安装和启动。

### 10.3 生产验证

- 使用 `seeker_ahua` 分别选择两种身份登录并验证数据隔离。
- 使用 `boss_zhang` 验证只能登录 Boss。
- 使用普通 seeker 验证不能登录 Boss。
- 同账号两台手机/两个浏览器验证同端互踢，电脑和手机同时在线不互踢。
- 验证私聊、投递查看、完整简历请求和安全下线通知。
- 检查 systemd、nginx、Redis、数据库、WebSocket 和应用日志无新增错误。

### 10.4 管理后台

- `super_admin` 可开通和禁用身份、查看会话并强制下线，操作产生完整审计日志。
- `auditor` 可审核企业/Boss 权限及投诉申诉，但不能越权修改管理员或执行无依据的敏感操作。
- `viewer` 所有写接口返回 403，敏感字段保持脱敏。
- 企业审核与 Boss 身份激活满足事务原子性，失败时全部回滚。
- 会话强退后旧 Token 立即失效，后台显示的新状态与 Redis 一致。
