# 微招系统精简完整接口测试手册（Word V2）设计说明

## 1. 目标

在保留现有 `API_TESTING.md`、`API_TESTING_COMPLETE.md`、`微招系统完整接口测试文档.docx` 和桌面旧文档的前提下，新建一份面向 Apifox/Postman 手工测试的 Word 手册。新版不展开冗长原理，每个接口都提供足以发起请求的完整数据，并覆盖当前代码中的全部 Controller 路由以及 WebSocket/STOMP 通信入口。

## 2. 覆盖范围与事实来源

接口清单以当前 `master` 代码为唯一事实来源：

1. 扫描 `src/main/java/com/weib/controller/**/*Controller.java` 的 `GET/POST/PUT/DELETE/PATCH` 映射。
2. 用生产 `/v3/api-docs` 交叉核对 REST 接口的方法、路径、参数和 Schema。
3. 从 Controller 方法签名、DTO、实体校验注解及安全配置补全 OpenAPI 未覆盖的表单接口、页面路由、Session/CSRF 规则和权限。
4. 单独读取 WebSocket 配置及消息 DTO，补充握手地址、订阅地址和发送目的地。

设计基线扫描结果为 158 个 MVC 路由，其中业务接口 130 个、页面路由 28 个；生成时必须重新扫描，最终数字以生成时的当前代码为准。任何代码接口未进入 Word，覆盖校验都应失败。

不作为接口条目的内容：静态 CSS/JS/图片、构建脚本、内部 Java 方法和数据库表。页面型 GET 路由收录在附录，避免与 JSON/表单业务接口混排。

## 4. 平台标注规则

每个接口标题前使用固定平台徽标：

- `📱 APP 专用`：所有 `/api/mobile/**` 接口。
- `Web/App 通用`：实际同时被 Web 和 Android 调用的公共接口。
- `Web 专用`：Session/Thymeleaf 表单或仅 Web 页面使用的接口。
- `管理后台`：所有 `/api/admin/**` 接口。
- `页面路由`：返回 HTML 的 GET 路由。
- `WebSocket`：握手、发送、订阅和在线状态通信入口。

App 专用章节使用蓝色章节带，接口卡片左侧显示蓝色 `📱 APP 专用` 徽标；目录和接口总览表也保留同样标识，不能只在正文出现一次。

## 5. 单接口固定模板

每个业务接口都采用相同的紧凑卡片，字段顺序固定：

1. **接口编号与平台徽标**：例如 `APP-AUTH-001｜📱 APP 专用`。
2. **功能**：一句话说明接口完成什么业务操作。
3. **请求方式与完整地址**：例如 `POST http://superorange.top/api/mobile/auth/login`。
4. **权限与前置条件**：匿名、Web Session、App Bearer、管理员 Bearer、角色要求、公司审核状态等。
5. **Content-Type 与请求头**：只列该接口实际需要的头。
6. **参数表**：位置、字段名、类型、必填、规则、可直接使用的示例值。
7. **可复制请求数据**：按接口实际类型提供完整 JSON、`x-www-form-urlencoded`、query string 或 multipart 字段清单。
8. **成功响应示例**：保留真实统一响应结构和关键业务字段。
9. **变量来源**：说明 `jobId`、`encodedId`、`applicationId`、`conversationId`、文件名等由哪个前置接口获得。
10. **关键错误**：只列该接口最常见的 2 至 4 个状态或业务错误，不重复通用长篇说明。

接口描述不得使用“同上”“参考页面”“按实际填写”代替请求数据。没有请求体的接口明确写“请求体：无”。

## 6. 可执行测试数据规则

- 基础地址默认使用 `http://superorange.top`。
- 提供求职者、招聘者和管理员测试变量区；最终生成前在生产环境验证账号可登录，再写入可用测试账号。密码只写用户已指定的统一测试密码，不读取或暴露 BCrypt 哈希、服务器密码或其他基础设施凭据。
- 验证码不能写固定值。手册提供完整顺序：先建立 Cookie Session，带账号密码参数请求验证码，人工读取验证码，再在有效期内提交登录。
- 动态 ID 不能伪造。每种 ID 都给出获取接口和示例提取路径，例如从职位列表响应取得 `data.content[0].id`。
- 写接口使用安全、可识别的练习数据，如标题前缀 `[API测试]`；涉及创建、收藏、点赞、投递和处罚时写明清理或撤销接口。
- `Idempotency-Key` 使用 `{{idempotencyKey}}`，并说明每次新业务请求生成新 UUID，重试同一请求复用原值。
- 文件上传提供 Windows 本地文件示例路径和允许的类型/大小，不在文档中嵌入真实个人文件。
- 对 `encodedId` 与数据库自增 ID 做明显区分，避免把两种 ID 混用。

## 7. Word 视觉与排版

采用 `compact_reference_guide` 设计预设，强调快速查找而不是装饰：

- Letter 纵向、1 英寸页边距、微软雅黑/Arial 正文字体。
- 深蓝标题、青绿色通用接口徽标、蓝色 App 徽标、紫色管理后台徽标。
- 接口标题保持与第一张信息表同页；跨页表格重复表头，不使用固定行高。
- 请求和响应使用 Consolas 等宽字体、浅灰底代码块，确保 JSON 可复制。
- 参数表使用固定列宽和足够单元格内边距，长文本自动换行。
- 页眉显示“微招系统接口测试手册 V2”，页脚显示页码和生成日期。

目标是紧凑但不拥挤；不采用“一接口一页”，允许同一页放置多个短接口卡片。

## 8. 生成方式

为避免 158 个接口的手工遗漏和格式漂移，先生成结构化接口清单，再由 Python 脚本统一构建 Word：

1. 从代码提取方法、路径、Controller 和页面/业务分类。
2. 为每个接口补充人工维护的功能、权限、参数、请求样例、响应样例和变量来源。
3. 生成 Word、目录字段、接口卡片、代码块和附录。
4. 生成覆盖证据 JSON，记录实现接口数、文档接口数、缺失项、未知项和 App 标注数。

生成器和覆盖校验脚本纳入仓库，后续接口变化时可重新生成 V2，而不是继续人工追加 Word 段落。

## 9. 验收标准

1. 旧版 Markdown 和 Word 文件内容及时间戳不被改写。
2. 当前全部 MVC 路由均在 Word 中出现一次且方法、路径准确；页面路由全部位于附录。
3. `/api/mobile/**` 接口全部带 `📱 APP 专用` 标记，且无非移动接口被误标为 App 专用。
4. 每个业务接口都包含功能、请求方式、完整 URL、权限、Content-Type、参数或“无参数”、可复制请求数据及成功响应。
5. 登录、验证码、CSRF、Session、Bearer、幂等和动态 ID 的前置链路可以独立完成，不依赖查看 Web 页面源码猜字段。
6. 抽样执行普通登录、App 登录、职位查询、简历查询、论坛查询和管理员查询，确认手册中的请求格式可被生产接口接受；写操作只在可安全清理时执行。
7. Word 通过结构审计、接口覆盖脚本和无障碍基础审计。
8. 使用 `render_docx.py` 渲染全部页面，并逐页检查缺字、溢出、表格截断、代码块跨页和异常空白；修复后重新渲染，直至通过。
